---
title: "维护规则"
description: "适合多人协作的最小维护规则。"
---
这套知识库可以按“轻量 wiki”来维护，而且很适合多人一起改。

最简单的做法就是：

- 用 Markdown 做正文
- 用固定文件夹存素材
- 用 Git 做版本记录

这样既方便编辑，也不容易把内容改丢。

## 推荐的维护分工

### 1. 产品 owner

负责：

- `00-product-profile.md`
- 红线、边界、SKU、基础事实

### 2. 内容 / 写稿同学

负责：

- `10-claim-cards/`
- `30-compare-cards/`
- `50-writing-recipes/`

### 3. 资料 / 运营同学

负责：

- `20-proof-cards/`
- `40-asset-registry/`

## 最低维护规则

- 改产品事实时，优先改主档，不要只改卖点卡
- 改卖点时，尽量说明适用边界
- 新增素材时，必须补到素材登记表
- 对比卡不要写成攻击竞品，重点写“怎么配合”
- 不确定的内容，宁可先标“待确认”，不要写死

## 推荐工具

如果你希望多人一起维护，完全可以先用最轻的方式：

- Obsidian：适合 Markdown 编辑和双链查看
- VS Code：适合批量编辑
- GitHub / GitLab：适合多人协作和版本记录

后面如果你想要更像 wiki 的体验，可以再上：

- Outline
- GitBook
- Wiki.js

但建议先把这套 Markdown 结构跑顺，再决定要不要套一层网页。

## 一个很实用的原则

先把这套库当成：

> “多人可编辑的 Markdown 知识库”

而不是先把它当成“必须上系统的 wiki”。

因为只要结构清楚，后面无论接 Obsidian、GitBook 还是内部网页，迁移都很轻。
